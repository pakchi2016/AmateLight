# ログ監査ミドルウェア 要件定義・基本設計書（ログインアクセス機能抜粋）

## 1\. システム要件（全体構想）

| 項目 | 内容 |
| :--- | :--- |
| **対象OS** | Windows, Linux |
| **開発言語** | C\# (.NET) |
| **アーキテクチャ** | 独立したマイクロサービス（将来的に他アプリとイベント駆動・gRPC等で連携想定） |

## 2\. 機能要件（ログインアクセス監視機能）

| 機能名 | 要件詳細 |
| :--- | :--- |
| **ログイン検知** | 対象OSの標準監査機構（Windows: セキュリティイベントログ, Linux: auditd/syslog）から、ログインイベントの「開始時刻」「アカウント名」「接続元（IP/端末名）」「セッション識別子」を取得する。 |
| **ログアウト検知** | 同監査機構からログアウトイベントを取得する。 |
| **セッション照合** | ログインとログアウトの各イベントを、OS固有のセッション識別子（ログオンID、auid等）を用いて**照合**する。 |
| **状態・時間管理** | ログイン検知時に「稼働中フラグ: true」として記録を開始し、ログアウト検知時に終了時刻の打刻と滞在時間（何時間何分）を算出し、「稼働中フラグ: false」に更新する。 |
| **エクスポート出力** | 単一テーブルの個別出力、および\*\*監査ログ（ファイル名）\*\*やネットワーク接続等の別イベントとマージした時系列ソート出力に対応する。 |

## 3\. データモデル・データベース設計

| 項目 | 内容 |
| :--- | :--- |
| **採用データベース** | SQLite（ローカルデータベース） |
| **テーブル1: LoginSessions** | ログインセッションの状態管理用テーブル。<br>【カラム例】 セッションID (PK), アカウント名, 接続元, 開始時刻, 終了時刻, 稼働中フラグ (true/false) |
| **テーブル2: AuditEvents** | 汎用イベント追記用テーブル。<br>【カラム例】 イベントID (PK), セッションID (FK), イベント種別, 操作詳細, 発生時刻 |

## 4\. 異常系仕様（OS強制終了時のフェイルセーフ）

| 項目 | 内容 |
| :--- | :--- |
| **補正トリガー** | ミドルウェア（本アプリ）の起動時。 |
| **シャットダウン時刻特定** | OSのシステムログ（Windows: EventID 6008等, Linux: journal/カーネルログ等）から、前回システムがダウンした正確な時刻を取得・特定する。 |
| **セッション強制クローズ** | 特定したシャットダウン時刻を、DB内で「稼働中フラグ: true」となっている全セッションのログアウト時刻として適用し、フラグを一斉に「false」に更新してデータの不整合を解消する。 |

## 5\. 未定義事項（今後の検討課題）

現時点で詳細な仕様が決定しておらず、今後定義が必要な項目は以下の通りですわ。

  * **DBへのデータ書き込み方式**: 事象発生の都度INSERT/UPDATEを行うか、メモリに蓄積して非同期でバルクインサートを行うか（パフォーマンス要件）：**未定義**
  * **エクスポートファイルの形式**: 出力するファイルの厳密なフォーマット（CSV, JSON等）および文字コード：**未定義**
  * **ログの保存ポリシー**: ローカルDB内のデータ保持期間、およびローテーションの有無：**未定義**
  * **資産台帳等の外部データ連携**: IPアドレスや端末名をわかりやすい名称に変換するための、**資産台帳**等との具体的な連携インターフェース：**未定義**
  * **UI/CLIの仕様**: ミドルウェア単体で稼働する際の、運用者向けインターフェースの詳細：**未定義**
